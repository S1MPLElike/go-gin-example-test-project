# go-gin-example API 接口测试计划

## 1. 测试目标

对 go-gin-example 项目的 RESTful API 进行接口级功能测试，覆盖正常流程、异常输入、边界值和鉴权场景，确保接口行为符合 [API 文档](API.markdown) 定义。

## 4. 测试策略

### 4.1 每接口覆盖维度

| 维度 | 说明 |
|------|------|
| 正常场景 | 必填字段合法，接口返回 200 |
| 必填校验 | 逐字段置空，断言 400 |
| 边界值 | 字符串长度刚好/超出限制、int 默认值/0/负数 |
| 鉴权 | 不带 Token、Token 伪造，断言 401 + 对应错误码 |
| 业务校验 | 关联数据不存在（如 tag_id 指向不存在的标签）、重复操作 |

### 4.2 测试数据管理

- **Token**：`session` 级别 fixture，整个测试会话登录一次，所有用例共享（[conftest.py](conftest.py#L10-L18)）
- **created_tag**：fixture 创建测试标签，用例结束后在 teardown 中软删除（[conftest.py](conftest.py#L25-L47)）
- **文章依赖标签**：`test_articles.py` 内置 `_create_tag()`、`_create_article()` 辅助方法，用例内部自建自用
- **随机化**：所有测试名称使用 `random.randint` 附加后缀，避免用例间数据冲突

### 4.3 不测试的内容

- 数据库事务回滚、连接池等底层行为
- Redis 缓存命中率、过期策略
- 高并发、压力测试
- `tags/export` 导出功能（涉及文件 I/O，需要额外环境准备）
- `articles/poster/generate` 海报生成（依赖外部字体和图片资源）

## 5. 测试框架与工具

| 工具 | 版本   | 用途 |
|-|------|------|
| pytest | 9.11 | 测试运行器 |
| allure-pytest | -    | 测试报告生成 |
| requests | -    | HTTP 请求发送 |
| random | 内置   | 测试数据随机化 |
| ||
配置文件 [pytest.ini](pytest.ini) 设置默认生成 Allure 结果到 `./allure-results` 目录。

## 6. 执行方式

```bash
# 执行全部测试
pytest

# 按模块执行
pytest test_api_folder/test_auth.py
pytest test_api_folder/test_tags.py
pytest test_api_folder/test_articles.py

# 生成 Allure 报告
allure serve .\allure-results
```

执行前需要确保 Go 服务已在 `127.0.0.1:8000` 启动，MySQL 和 Redis 可用，测试账号 `test/test123` 已注册。

## 7. 缺陷管理

测试过程中发现的 Bug 将记录在测试报告中，核心问题包括：

| 编号 | 模块 | 问题描述 | 严重程度 |
|------|------|----------|----------|
| BUG-01 | 标签-删除 | `DeleteTag` 校验失败分支缺少 `return`，导致重复写响应体，返回非法 JSON | 高 |
| BUG-02 | 标签-删除 | 删除标签后未清除 Redis 缓存，导致列表接口仍然返回已删除数据 | 高 |
