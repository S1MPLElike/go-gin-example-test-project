读操作：先查缓存，缓存未命中再查数据库，然后写入缓存
写操作：先更新数据库，然后删除缓存（或更新缓存）

## Bug 深度解析：DeleteTag 缺少 `return` 导致重复写响应

### 一、问题本质

看源码 [tag.go](file:///c:/Users/20743/PycharmProjects/go_gin_exampleApi_test/go-gin-example/routers/api/v1/tag.go#L169-L198)：

```go
if valid.HasErrors() {
    app.MarkErrors(valid.Errors)
    appG.Response(http.StatusBadRequest, e.INVALID_PARAMS, nil)
    // ← 这里缺少 return！
}

tagService := tag_service.Tag{ID: id}
exists, err := tagService.ExistByID()
// ... 继续执行到后面，又写了一个 JSON
```

当 ID 为负数（`-1`）或非数字（`abc` 转成 `0`）时：
1. `valid.Min(id, 1)` 校验失败 → 写入第一个 JSON `{"code":400,...}`
2. 代码继续往下走 → 调用 `ExistByID()` 查不到记录 → 写入第二个 JSON `{"code":10003,...}`

最终响应体是两个 JSON 拼在一起：
```
{"code":400,"msg":"请求参数错误","data":null}{"code":10003,"msg":"该标签不存在","data":null}
```

### 二、可讲的点

#### 1. 如何发现的？

> "我写了删除标签的测试用例，覆盖了 ID 为负数和非数字的场景。运行时发现 `resp.json()` 抛出 `JSONDecodeError: Extra data`。我打印了 `resp.text` 发现返回了两个 JSON 对象拼在一起，这明显不符合 API 契约。
>
> 于是我去看 `DeleteTag` 的源码，发现 `valid.HasErrors()` 分支后没有 `return`，导致代码继续执行到后面的逻辑，又写了一次响应。"

#### 2. 为什么这是一个严重的 bug？

> "这个 bug 的严重程度不低：
>
> **影响面广**：所有参数校验失败的场景都会返回非法 JSON
> - ID 为负数、0、非数字
> - 任何需要参数校验的接口如果也有这个问题，都会受影响
>
> **客户端崩溃风险**：大多数客户端（包括我写测试用例用的 requests 库）都会把非标准 JSON 当成解析错误处理。用户调用删除接口时，如果传了非法参数，客户端可能直接抛异常，而不是收到友好的错误提示。
>
> **数据不一致**：虽然这里只是返回错误，但如果是写操作（比如更新），缺少 `return` 可能导致先返回错误响应，然后继续执行数据库操作，造成数据变更但客户端不知道。"

#### 3. 修复方案

> "修复很简单：在 `appG.Response(...)` 后面加 `return`。但更重要的是思考为什么会出现这个问题——这是一个典型的**代码审查遗漏**。
>
> 更彻底的做法是在团队中建立代码规范：所有 `appG.Response()` 调用后必须跟 `return`，或者用 `c.Abort()` 来终止请求处理链。"

#### 4. 如何验证修复？

> "修复后，我会：
> - 重新运行之前失败的测试用例，确认能正常解析 JSON，且 code 返回 400
> - 检查其他接口是否有类似问题（比如 `EditTag`、`DeleteArticle`）
> - 建议团队做一次全局搜索，检查所有 `appG.Response` 或 `c.JSON` 调用后是否都有正确的终止逻辑"

#### 5. 举一反三：同类问题的预防

> "这个问题暴露了一个开发习惯问题。在 Gin 框架中，中间件和 handler 的设计要求：一旦写入响应，就应该调用 `c.Abort()` 或 `return` 终止后续处理。
>
> 我建议：
> - 在代码审查 checklist 中加入一条：'检查所有响应写入后是否有正确的终止逻辑'
> - 可以考虑封装一个统一的响应函数，自动调用 `c.Abort()`
> - 使用静态分析工具（如 golangci-lint）来检测这类问题"
