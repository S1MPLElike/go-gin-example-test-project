## 用例 1：正常场景 - 创建标签

| 项目 | 内容                                                                       |
|------|--------------------------------------------------------------------------|
| **编号** | TC-TAG-001                                                               |
| **模块** | 标签                                                                       |
| **标题** | 正常创建标签——必填字段合法                                                           |

## 用例 2：业务规则 - 标签名不可重复

| 项目 | 内容                                           |
|------|----------------------------------------------|
| **编号** | TC-TAG-002                                   |
| **模块** | 标签                                           |
| **标题** | 创建标签——名称已存在                                  |
| **前置条件** | 已存在同名标签                                      |
| **测试数据** | 第一次：`name="test_dup_{random}"`，第二次：相同 `name` |
| **步骤** | 先创建一个标签，再用相同 `name` 创建第二个                    |

**设计思路**：不依赖数据库中已有数据，而是**用例内部完成数据准备**——先创建、再尝试重复创建。这种"自给自足"的模式避免了数据库状态对测试结果的影响。

## 用例 4：组合筛选 - 按 name + state 查标签列表

| 项目 | 内容 |
|------|------|
| **编号** | TC-TAG-004 |
| **模块** | 标签 |
| **标题** | 获取标签列表——name + state 组合筛选 |
| **前置条件** | 已创建一个启用状态的标签 |
| **测试数据** | 创建时：`name="combo_filter_{random}"`，`state=1`；查询参数：`name=xxx&state=1` |
| **步骤** | GET `/api/v1/tags?name=xxx&state=1` |
| **预期** | 返回的列表中每条记录的 name 和 state 都匹配筛选条件 |
| **后置处理** | fixture内有自动清理机制  |

**设计思路**：单个筛选条件测试比较简单，但**组合筛选**能验证后台 SQL 的 WHERE 条件拼接是否正确、Redis 缓存 key 的生成是否包含了所有筛选参数。这个用例同时断言了"过滤后的数据都符合条件"而非只看 total。

## 用例 5：数据一致性 - 删除后列表查不到（发现 Bug）

| 项目 | 内容 |
|------|------|
| **编号** | TC-TAG-005 |
| **模块** | 标签 |
| **标题** | 验证软删除——删除后 GET 列表 total=0 |
| **前置条件** | 已创建一个标签，记录其 name |
| **测试数据** | `_create_tag()` 自动生成随机 name，`state=1` |
| **步骤** | ① DELETE 删除该标签 ② GET `/api/v1/tags?name=xxx` |
| **预期** | 删除后 total=0，lists 为空 |
| **实际结果** | FAILED——total=1，因为 Redis 缓存未清除 |
| **后置处理** | 无（标签已在步骤①被软删除，无需额外清理） |

**设计思路**：这个用例不仅验证接口功能，还验证了**数据一致性**。通过"创建 → 删除 → 查询"的完整链路，发现了一个真实 Bug：源码中 `DeleteTag` 只做了数据库软删除，但没有清除 Redis 缓存，导致列表接口在缓存过期前仍然返回已删除的数据。这是一个典型的 Cache-Aside 模式实现漏洞。

> 面试可讲：发现这个 Bug 的过程、为什么会出现、修复方案是什么。

## 用例 6：跨模块业务校验 - tag_id 指向不存在的标签

| 项目 | 内容 |
|------|------|
| **编号** | TC-ART-001 |
| **模块** | 文章 |
| **标题** | 创建文章——tag_id 指向不存在的标签 |
| **前置条件** | 已登录（无需预先创建标签） |
| **测试数据** | `tag_id=99999`（固定不存在的 ID），title/desc/content 等字段为正常随机值 |
| **步骤** | POST `/api/v1/articles?token=xxx`，`tag_id=99999` |
| **预期** | code=10003（"该标签不存在"） |
| **后置处理** | 无（创建失败，数据库无新记录） |

**设计思路**：文章依赖标签，创建文章时必须验证 `tag_id` 的有效性。这个用例覆盖了**跨模块的业务依赖校验**——不只看自己接口的参数校验，还要验证关联数据的完整性。面试时可以说"文章模块 14 个用例中有 4 个涉及标签依赖校验，包括 tag_id=0、负数、不存在三种场景"。

## 用例 7：必填校验 - title 为空

| 项目 | 内容 |
|------|------|
| **编号** | TC-ART-002 |
| **模块** | 文章 |
| **标题** | 创建文章——title 为空 |
| **前置条件** | 已登录，已创建一个有效标签 |
| **测试数据** | `title=""`（空字符串），其他字段（tag_id/desc/content/created_by/cover_image_url）为正常随机值 |
| **步骤** | POST `/api/v1/articles?token=xxx`，title 传空字符串 |
| **预期** | HTTP 400，code=400 |
| **后置处理** |fixture内有自动清理机制  |

**设计思路**：创建文章有 6 个必填字段（`tag_id`、`title`、`desc`、`content`、`created_by`、`cover_image_url`），我**逐字段各写了一个校验用例**，而不是一个用例把所有字段置空。这样做的好处：① 失败时能精确定位到哪个字段的校验有问题；② 符合"一个用例只测一个点"的原则。

## 用例 8：参数类型 - ID 为负数

| 项目 | 内容 |
|------|------|
| **编号** | TC-ART-003 |
| **模块** | 文章 |
| **标题** | 更新文章——ID 为负数 |
| **前置条件** | 已登录，已创建一个有效标签 |
| **测试数据** | URL 路径中的 `id=-1`，表单字段（tag_id/title/desc 等）为正常随机值 |
| **步骤** | PUT `/api/v1/articles/-1?token=xxx` |
| **预期** | HTTP 400，code=400 |

**设计思路**：URL 路径参数不像表单参数那样容易被校验框架拦截，需要验证 `com.StrTo().MustInt()` 将非数字转为 0、负数不被 `Min(1)` 接受的情况。同时这个用例覆盖了**负数、0、非数字**三类非法 ID 输入，构成参数类型校验的组合。

## 用例 9：重复操作 - 两次删除同一篇文章

| 项目 | 内容 |
|------|------|
| **编号** | TC-ART-004 |
| **模块** | 文章 |
| **标题** | 删除文章——重复删除同一个 ID |
| **前置条件** | 已创建一篇文章（通过 `_create_article()` 自动创建） |
| **测试数据** | 由 `_create_article()` 自动生成的随机文章（含随机 title/desc/content，关联有效 tag_id） |
| **步骤** | ① DELETE 删除该文章 ② 再次 DELETE 同一 ID |
| **预期** | 第一次 code=200；第二次 code=10011（"该文章不存在"） |
| **后置处理** | 无（文章已在步骤①被删除，第二次调用不会产生副作用） |

**设计思路**：验证接口的**重复操作错误处理**。很多接口只测"创建→删除"就结束了，但连续两次删除能否正确处理第二次，往往被忽略。这个用例也间接验证了删除操作是否真的是软删除（第二次会因 `ExistByID` 返回 false 而报 10011）。

## 用例 10：鉴权 - 不带 Token 调用需认证的接口

| 项目 | 内容 |
|------|------|
| **编号** | TC-AUTH-001 |
| **模块** | 标签 |
| **标题** | 创建标签——不带 Token |
| **前置条件** | 无（不登录，不依赖任何 Token） |
| **测试数据** | `name="test_tag_{random}"`，`created_by="test_admin"`，`state=1`；URL **不带** `?token=xxx` |
| **步骤** | POST `/api/v1/tags`（不带 ?token=xxx） |
| **预期** | HTTP 401，code=400（JWT 中间件返回的"请求参数错误"） |
| **后置处理** | 无（请求被中间件拦截，不会创建任何数据） |

**设计思路**：JWT 鉴权是所有业务接口的基础，每个接口我都覆盖了"不带 Token"和"Token 伪造"两个场景。这里用**标签接口**作为代表，展示鉴权测试的标准模式：不带 Token → 401 + code=400；伪造 Token → 401 + code=20001。用例不依赖 `auth_url` fixture，直接调裸 URL。
