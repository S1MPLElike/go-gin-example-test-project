## 3. 缺陷记录

### BUG-01：DeleteTag 校验失败后缺少 return，导致重复写响应

| 项目 | 内容 |
|------|------|
| **严重程度** | 高 |
| **所属模块** | 标签 |
| **接口** | `DELETE /api/v1/tags/:id` |
| **触发条件** | 传入负数 ID（-1）或非数字 ID（abc） |
| **现象** | 响应体返回两个 JSON 拼接：`{"code":400,...}{"code":10003,...}`，客户端 JSON 解析报 `Extra data` 错误 |
| **根因** | `routers/api/v1/tag.go` 的 `DeleteTag` 函数中，`valid.HasErrors()` 分支调用 `appG.Response()` 后缺少 `return`，导致代码继续执行到 `ExistByID` 逻辑，再次写入了响应 |
| **影响** | 所有参数校验失败的场景都返回非法 JSON，客户端无法正常解析 |
| **修复** | 在 `appG.Response(http.StatusOK, e.INVALID_PARAMS, nil)` 后加 `return` |

### BUG-02：删除标签后未清除 Redis 缓存，列表接口返回脏数据

| 项目 | 内容 |
|------|------|
| **严重程度** | 高 |
| **所属模块** | 标签 |
| **接口** | `DELETE /api/v1/tags/:id` → `GET /api/v1/tags` |
| **触发条件** | 创建标签 → 查询列表（缓存写入） → 删除标签 → 再次查询列表 |
| **现象** | 删除标签后，GET 列表接口仍返回已删除的标签数据（total=1），直到缓存 3600s 过期 |
| **根因** | `service/tag_service/tag.go` 的 `DeleteTag` 只做了数据库软删除，没有清除 Redis 中对应的缓存 key。而 `GetAll` 优先读缓存，导致返回脏数据 |
| **影响** | 用户删除标签后，列表页在缓存过期前仍显示已删除的标签，属典型 Cache-Aside 模式实现漏洞 |
| **修复** | 在 `DeleteTag` 删除成功后调用 `gredis.Delete(key)` 清除相关缓存；同理，`EditTag` 更新标签后也需清除缓存 |

## 4. 测试结论与建议

### 4.1 结论

- **认证模块和文章模块**共 44 个用例全部通过，接口功能正常
- **标签模块** 34 个用例通过 31 个，针对删除功能发现 2 个 Bug
- 本次测试共覆盖 3 个模块、10 个接口、78 个用例，通过率 96.15%，达到了接口功能测试的覆盖目标

### 4.2 建议

1. BUG-01 缺少 `return` 是简单但严重的编码疏漏，建议同步检查其他接口是否存在相同问题
2. BUG-02 属 Cache-Aside 模式实现漏洞，修复时需同步检查 `EditTag` 的缓存清除逻辑
3. 两个 Bug 修复后，重新运行标签模块 34 个用例进行回归验证
4. 后续可扩展：为所有写操作统一增加 Redis 缓存清除机制，防止类似问题再次出现
